# 播放队列来源补全设计

## 背景

当前仓库里，页面歌曲列表和播放队列存在耦合：

- 左侧页面列表会按分页或滚动加载逐步追加数据。
- `usePlaybackStore.queue` 在部分场景下直接基于页面当前已加载列表初始化。
- 如果来源列表很大，而页面只加载了前几十首，播放器就可能只在这部分歌曲里循环。
- 播放列表抽屉虽然已经有队列补全和缓存能力，但触发时机仍偏 UI 导向，不能保证“不打开抽屉也不断歌”。

目标是把“浏览列表”和“播放队列”彻底解耦，让播放系统只认一份权威播放队列。

## 目标

- 左侧页面列表继续按页面体验需要分页加载，不承担播放器完整性责任。
- 单一来源播放时，点击后可以快速起播，但播放队列最终必须补全为完整来源队列。
- 不打开播放列表抽屉时，播放也不能因为页面未滚动加载而断歌。
- 打开抽屉时不重复发第二套全量请求，复用同一个来源缓存或进行中的补全任务。
- `next / previous / repeat / shuffle` 一律基于权威播放队列运行。

## 非目标

- 不改搜索、本地文件、手动追加、混合队列的基础播放语义。
- 不在本次设计里引入拖拽排序、队列持久化到磁盘、跨会话完整队列缓存。
- 不让页面列表的滚动状态反向驱动播放队列内容。

## 方案对比

### 方案 A：页面分页列表与播放队列完全解耦，起播后后台补全完整来源队列

这是推荐方案。

- 页面列表只做浏览，按 `30` 或 `50` 条滚动分页。
- 开始播放单一来源时，只拉首批较小数据量立即起播。
- 起播成功后，后台继续补全完整来源队列，并持续同步到 `usePlaybackStore.queue`。
- 抽屉只展示和操作这份权威播放队列，打开时复用同一补全任务或缓存。

优点：

- 页面首屏和滚动体验最好。
- 不依赖用户打开抽屉，也不会断歌。
- 播放逻辑、抽屉展示、快捷键切歌都统一到同一份队列。

缺点：

- 播放入口需要按来源类型声明补全策略。
- 需要维护来源级缓存和 inflight 任务去重。

### 方案 B：只有抽屉首次打开时才拉完整来源队列

不推荐。

- 页面继续分页。
- 抽屉默认不拉完整队列。
- 只有用户首次打开抽屉时，才分批拉完整歌单并覆盖当前播放队列。

问题：

- 如果用户从不打开抽屉，播放队列就永远不完整。
- 播到当前已加载列表末尾时，仍可能断歌或只在局部循环。

### 方案 C：左侧页面一进入就直接拉完整来源列表

不推荐。

- 页面和播放队列直接共用一份完整列表。

问题：

- 大列表首屏会变慢。
- 页面承担了不属于它的播放器职责。
- 与当前仓库已经落地的滚动分页方向冲突。

## 推荐设计

采用方案 A。

### 1. 权威播放状态

播放器只认 `usePlaybackStore` 里的状态：

- `queue`
- `queueSourceKey`
- `currentIndex`
- `currentTrack`
- `playbackMode`

其中：

- `queue` 是唯一权威播放队列。
- `currentIndex` 是唯一权威当前位置。
- 抽屉、上一首、下一首、循环、随机全部基于这组状态运行。

页面自己的分页列表不再决定播放器边界。

### 2. 来源分类

按“是否能稳定映射到单一完整来源”分两类。

#### 单一来源

- `playlist:<id>`
- `liked-songs:<playlistId>`
- `album:<id>`

这三类统一走“首批起播 + 后台补全完整队列”。

#### 非单一来源

- `search`
- `local`
- `manual`
- `mixed`

这几类不做后台补全，当前传入的队列就是最终权威队列。

### 3. 请求与补全时序

以歌单或“我喜欢的音乐”为例：

1. 用户点击播放。
2. 页面或入口先请求首批歌曲，例如 `limit=100`。
3. 拿到首批后立即调用 `playQueueFromIndex(firstBatch, startIndex, sourceKey)` 开始播放。
4. 起播成功后，后台任务从 `offset=100` 开始继续补全完整来源队列。
5. 后台补全按接口限制分批拉取，例如 `limit=1000`：
   - 第一次：`offset=100, limit=1000`
   - 第二次：`offset=1100, limit=1000`
6. 每补到一页，就把累计结果通过 `syncQueueFromSource(sourceKey, accumulatedQueue)` 扩容当前播放队列。
7. 如果用户此时打开抽屉，抽屉只读取缓存或复用正在进行的补全任务，不再另发一套全量请求。

这样可以同时满足：

- 点击马上播。
- 不打开抽屉也不会断歌。
- 打开抽屉不会重复请求。

### 4. 缓存与任务复用

来源级缓存按 `queueSourceKey` 维度维护，不能只用一个全局布尔值。

`playback-queue-hydration.model.ts` 负责维护两类状态：

- 已补全完成的完整队列缓存。
- 正在进行中的来源补全任务。

要求：

- 同一来源重复触发补全时，直接复用 inflight 任务。
- 同一来源完整队列已缓存时，直接复用缓存。
- 切换到新来源时，旧来源缓存不影响新来源。

### 5. 数据模型

完整播放队列不是“只存歌曲 id”，而是存项目当前播放与展示所需的轻量播放模型。

至少保留：

- `id`
- `name`
- `artistNames`
- `albumName`
- `coverUrl`
- `duration`
- `sourceUrl`（仅本地或特殊来源需要时保留）

这样可以保证：

- 抽屉展示足够完整。
- 当前底部播放条和播放器页不需要再额外回查大部分基础信息。

### 6. UI 行为

#### 左侧页面列表

- 继续按页面体验需求分页加载。
- 只负责浏览、定位和高亮当前歌曲。
- 不负责维护完整播放队列。

#### 右侧播放列表抽屉

- 只展示 `usePlaybackStore.queue`。
- 使用虚拟滚动承载上千首条目。
- 打开时若当前来源支持补全，则仅确保 hydration 已启动或已命中缓存。
- 关闭时不清空缓存。

#### 高亮同步

- 左侧列表和右侧抽屉都基于当前播放状态同步高亮。
- 播放器推进依赖 `currentIndex`，UI 高亮可以使用 `currentTrack.id` 辅助判断。

## 模块边界

### `src/renderer/stores/playback-store.ts`

职责：

- 保存权威播放状态。
- 提供切歌、模式切换、队列同步能力。
- 不直接负责来源接口请求与分批补全。

### `src/renderer/model/playback-queue-hydration.model.ts`

职责：

- 按来源发起后台补全。
- 维护缓存和 inflight 任务。
- 将增量结果同步回 `usePlaybackStore.queue`。

### 页面播放入口

职责：

- 判断当前入口属于哪类来源。
- 获取首批起播数据。
- 传入正确的 `queueSourceKey`。
- 对单一来源启动 hydration。

### `PlaybackQueueDrawer`

职责：

- 展示权威播放队列。
- 响应点击切歌。
- 在需要时确保当前来源的补全任务已存在。

## 错误处理

- 单一来源补全失败时，不中断已开始的前序播放。
- 抽屉可继续展示当前已知队列。
- 补全失败只影响完整队列扩容，不影响当前已成功起播的歌曲播放。
- 补全任务应避免因为队列已切到别的来源而继续污染当前播放队列。

## 测试策略

至少覆盖以下行为：

- 单一来源入口首批起播后，会后台继续补全完整队列。
- 补全过程中，`syncQueueFromSource` 只扩容当前同源队列，不污染其他来源。
- 同一来源多处触发补全时，只复用一个 inflight 任务。
- 抽屉打开时命中缓存不重复请求。
- 抽屉点击切歌不重建整份队列。
- 非单一来源不会误触发完整来源补全。

## 实施顺序

1. 统一 `playlist / liked-songs / album` 的来源 key 规则。
2. 抽象并稳定 `playback-queue-hydration.model.ts` 的缓存与 inflight 复用。
3. 把这三类入口统一改成“首批起播 + 后台补全”。
4. 收口抽屉逻辑，只消费权威播放队列。
5. 其余入口显式标记为 `manual / local / search / mixed`，保持当前给定队列语义。
