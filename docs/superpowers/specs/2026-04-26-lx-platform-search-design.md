# LX 多平台分栏搜索设计

## 背景

当前项目已经接入 LX 音源能力，并支持以下平台解析：

- 网易云 `wy`
- QQ 音乐 `tx`
- 酷我音乐 `kw`
- 酷狗音乐 `kg`
- 咪咕音乐 `mg`

现有搜索入口仍然以统一结果流为主，不足以表达“用户明确要搜某个平台”的意图。与此同时，LX 音源还支持音质配置，现阶段缺少一套清晰的搜索期音质覆盖方案。

这次改造的目标不是新增第二个搜索系统，而是在**现有全局搜索弹窗**内，把 LX 多平台能力收成一套清晰、稳定、可扩展的**增强态交互模型**。

## 目标

### 业务目标

1. 在现有搜索弹窗中新增平台分栏搜索能力。
2. 搜索支持平台维度切换：`网易 / QQ / 酷我 / 酷狗 / 咪咕`。
3. 支持“默认音质 + 当前搜索会话临时音质覆盖”。
4. 在某个平台结果中点歌后，当前歌曲及由此生成的队列应**锁定该平台优先解析**。
5. 继续兼容当前已实现的 VIP / 非 VIP `fee` 分流逻辑，不绕过现有付费判断。
6. 只有在**已导入并启用 LX 音源**时，才显示多平台搜索增强态。

### 体验目标

1. 不新增第二搜索入口，继续沿用现有搜索弹窗。
2. 平台切换语义明确，不让用户在混合搜索结果里猜来源。
3. 搜索行为尽量轻量，首版不做五个平台并发搜索。
4. 切换平台时优先使用缓存结果，避免不必要的重复请求。
5. 当前搜索会话的音质调整不污染全局设置。

## 非目标

本次不做以下内容：

1. 不新增独立的 LX 搜索页或路由页。
2. 不做五个平台同时并发搜索。
3. 不把 LX 平台搜索和现有在线平台搜索做混合排序聚合。
4. 不把搜索会话的临时音质保存为长期配置。
5. 不绕过现有播放器解析链和 `fee` 分流策略。
6. 不修改本地乐库搜索结构。
7. 不在未导入 LX 音源时展示禁用态平台 tabs。

## 方案概览

### 核心方案

在现有全局搜索弹窗内，按能力分成两种状态：

### 默认态

当用户**未导入或未启用 LX 音源**时：

1. 保持当前单一搜索结构
2. 不显示平台 tabs
3. 不显示 LX 会话音质选择器
4. 不引入平台锁定播放语义

### LX 增强态

当用户**已导入并启用 LX 音源**时，在现有全局搜索弹窗内新增：

1. 平台 Tabs
2. 会话级音质选择器
3. 平台独立结果状态缓存
4. 锁平台播放上下文

增强态下，搜索时只查询当前平台，不并发查询其他平台。结果按平台缓存，切换平台时优先读缓存。

增强态下，播放时不再只依赖当前 UI 状态，而是把 `lockedPlatform` 和 `sessionQuality` 透传到播放队列上下文中。这样即使关闭搜索弹窗，后续队列仍然知道自己来自哪个平台。

### 为什么不用统一结果流

统一结果流的主要问题是：

1. 平台来源不明确，和 LX 的多平台能力天然冲突。
2. 音质覆盖会变得模糊，不清楚当前临时音质到底作用于哪条结果。
3. 点击某首歌后如果需要锁平台，统一流会把队列来源语义变得混乱。

因此本次明确采用**平台分栏搜索**。

## 信息架构

### 搜索弹窗头部结构

LX 增强态下，搜索弹窗头部调整为三层或两行布局：

1. 搜索输入框
2. 平台 Tabs：`网易 / QQ / 酷我 / 酷狗 / 咪咕`
3. 音质选择器：当前搜索会话临时音质

建议布局：

- 第一行：搜索输入框
- 第二行：左侧平台 Tabs，右侧音质选择器

如果当前弹窗空间不足，可做成：

- 第一行：搜索输入框 + 音质选择器
- 第二行：平台 Tabs

视觉重点应保持在“关键词输入”和“平台切换”，音质选择器作为次级控制。

### 搜索结果区

LX 增强态下，结果区只展示**当前选中平台**的结果。

每个平台独立维护：

- `items`
- `loading`
- `error`
- `hasSearched`
- `keywordSnapshot`

这样切换 tab 时：

1. 如果该平台已有当前关键词的缓存，直接显示缓存。
2. 如果没有，则触发查询。

## 状态模型

### 全局设置

在设置页新增或复用一个 LX 搜索/解析默认音质配置：

- `defaultLxSearchQuality`

这个配置是全局长期值，用于搜索弹窗打开时初始化会话音质。

### 搜索会话状态

LX 增强态下，搜索弹窗内部新增一份会话级状态：

- `keyword`
- `debouncedKeyword`
- `activePlatform`
- `sessionQuality`
- `platformResults`

其中：

- `sessionQuality` 关闭弹窗后销毁
- `platformResults` 只存在当前弹窗生命周期内

### 平台结果状态

LX 增强态下，每个平台维护如下结构：

```ts
type SearchPlatformState = {
  keywordSnapshot: string
  items: SearchResultItem[]
  loading: boolean
  error: string | null
  hasSearched: boolean
}
```

这样可以明确判断：

1. 当前 tab 是否查询过
2. 当前结果是否匹配当前关键词
3. 当前是否需要重新查询

## 查询流程

### 基本流程

默认态下，继续沿用当前单一搜索查询流程。

LX 增强态下：

1. 用户输入关键词
2. `debounce` 生效后得到 `debouncedKeyword`
3. 基于 `activePlatform + debouncedKeyword` 决定是否触发查询
4. 只查询当前平台
5. 查询结果写入当前平台缓存

### 平台切换流程

只有在 LX 增强态下才存在平台切换流程：

1. 用户切换平台 tab
2. 检查目标平台缓存：
   - 若 `keywordSnapshot === debouncedKeyword` 且已有结果，直接显示
   - 否则发起新查询

### 空关键词策略

当 `debouncedKeyword` 为空时：

- 不发起平台搜索请求
- 当前结果区回到初始状态或最近一次结果缓存显示前的空态

首版建议回到明确空态，不显示陈旧结果。

## 音质策略

### 优先级

音质优先级固定为：

1. 搜索会话临时音质 `sessionQuality`
2. 设置页默认音质 `defaultLxSearchQuality`

### 会话覆盖规则

LX 增强态下，搜索弹窗内修改音质后：

- 仅影响当前搜索会话的查询、点播和生成队列
- 不写回设置页全局默认值

### 查询与播放的音质传递

LX 增强态下，当前会话音质需要透传到：

1. 搜索结果点播
2. 点播后生成的播放队列
3. 由搜索结果触发的解析上下文

这样用户在当前会话里选定 `FLAC` 或 `320k` 后，播放行为才能保持一致。

## 播放与队列锁平台

### 锁平台规则

只有在 LX 增强态下，当用户在某个平台 tab 下点击一首歌播放时：

1. 当前歌曲解析上下文写入 `lockedPlatform`
2. 该次生成的队列也写入同样的 `lockedPlatform`
3. 后续队列内歌曲继续优先使用该平台解析

这条规则只针对**由搜索结果生成的播放上下文**。

### 为什么必须进队列上下文

不能只把平台锁保存在搜索弹窗组件状态中，因为：

1. 搜索弹窗关闭后状态会消失
2. 播放器和队列仍然要继续知道歌曲来自哪个平台

因此平台锁必须进入 `PlaybackTrack` 或等价的队列元数据结构。

### 与现有付费分流的关系

锁平台只决定“优先来源”，不覆盖现有付费规则。

现有 `fee` 分流逻辑继续生效：

- VIP：官方优先
- 非 VIP 且 `fee===0`：官方优先
- 非 VIP 且 `fee!==0`：跳过官方，走非官方解析链

也就是说：

1. `lockedPlatform` 只影响 LX 多平台解析顺序和目标平台
2. 不改变 `fee` 驱动的官方/非官方分流策略

## UI 细节

### 能力门控

搜索弹窗根据“是否已导入并启用 LX 音源”切两套 UI：

1. 默认态：
   - 沿用现有搜索 UI
   - 不展示平台 tabs
   - 不展示 LX 音质选择器
2. LX 增强态：
   - 展示平台 tabs
   - 展示会话级音质选择器
   - 展示平台来源 badge

不做“常驻显示但禁用”的平台 tabs，避免给用户错误心智。

### 平台 Tabs

LX 增强态下，平台 tabs 使用轻量 pill/tab 样式，文案固定为：

- 网易
- QQ
- 酷我
- 酷狗
- 咪咕

不在首版显示平台图标。

### 音质选择器

LX 增强态下，音质选择器使用紧凑型下拉：

- 128k
- 192k
- 320k
- FLAC
- Hi-Res / 24bit（如果项目已有统一命名则沿用）

若某个平台最终不支持当前音质，仍允许先带着该音质进入解析链，后续由解析侧做降级映射，不在 UI 层提前做平台差异分叉。

### 搜索结果项

LX 增强态下，每条结果至少展示：

- 歌名
- 歌手
- 专辑
- 平台来源 badge

来源 badge 既用于平台辨识，也为后续诊断“当前队列锁在哪个平台”提供一致视觉语言。

## 错误处理

### 平台级错误

LX 增强态下，某个平台搜索失败时：

- 只影响当前平台 tab
- 不污染其他平台状态
- 错误区提供“重试当前平台搜索”

### 无结果

LX 增强态下，当某个平台查询成功但结果为空：

- 展示该平台无结果空态
- 不自动切换到其他平台

这样用户能明确知道“这个平台搜不到”，而不是被系统暗中改平台。

## 性能策略

### 首版原则

1. 不做五平台并发搜索
2. 单平台查询 + 缓存复用
3. 搜索输入沿用已有防抖策略
4. 只有在 LX 增强态下才初始化平台级搜索状态

### 原因

五平台并发搜索虽然看起来响应更快，但会明显放大：

- 请求量
- 状态管理复杂度
- UI 同步复杂度

首版优先保证行为清晰和稳定，不为了表面“快”引入更高复杂度。

## 实施边界

### 本次必须实现

1. 现有全局搜索弹窗内的平台 tabs
2. 会话级音质覆盖
3. 单平台查询与缓存
4. 点歌锁平台并进入播放队列上下文
5. 与现有 `fee` 分流兼容
6. 根据 LX 音源启用状态切默认态 / 增强态两套 UI

### 本次不做

1. 五平台并发搜索
2. 平台搜索结果混合排序
3. 搜索结果多选操作
4. 平台级历史记录
5. 平台音质能力在搜索 UI 层做细粒度差异提示

## 风险与注意事项

### 风险 1：状态污染

如果继续沿用统一结果状态，不按平台拆缓存，后续切换平台和临时音质时会互相覆盖，造成结果错乱。

### 风险 2：播放平台上下文丢失

如果只把锁平台存在搜索弹窗里，关闭弹窗后后续队列会回到全局策略，行为不一致。

### 风险 3：绕过现有付费逻辑

如果在搜索播放链里直接硬写 LX 解析优先，而不走现有 `fee` 分流，会把你刚做完的 VIP / 非 VIP 策略打回去。

本次设计明确要求：

- 平台锁只附加在现有解析上下文中
- 不直接绕开 `fee` 规则

## 推荐实现顺序

1. 搜索弹窗状态模型重构
2. 平台 tabs 与音质选择器 UI
3. 单平台查询与缓存
4. 播放队列上下文增加 `lockedPlatform` 与会话音质
5. 解析链接入平台锁
6. 回归验证 `fee` 分流与普通搜索
